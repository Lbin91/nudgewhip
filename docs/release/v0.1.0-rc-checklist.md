## 1. Purpose

- 이 문서는 `docs/release/release-readiness-checklist.md`의 게이트를 실제 배포 순서로 압축한 실행 문서다.

## 2. Release Lane

- Release type: `First public beta`
- Distribution lane: `source-first GitHub release`
- Product scope: `macOS-only`, `Free core loop only`
- Out of scope for `0.1.0`:
  - `iOS companion`
  - `CloudKit sync`
  - `Pro packaging`
  - advanced whitelist / richer exception workflows beyond current beta promise

### Messaging rule

- README / GitHub release note에서는 `public beta` 표현을 유지할 수 있다.
- App Store metadata, social copy, email copy를 작성할 경우 `beta` 대신 `Early Access` 또는 `Preview`를 사용한다.

## 6. Execution Sequence

### Step 1. Freeze release scope

- Lock `v0.1.0` to the current README beta promise only.
- Remove or hide any surface that implies:
  - iPhone companion availability
  - CloudKit dependency
  - Pro entitlement shipping now
  - real-time sync guarantees

Evidence:

- README scope section
- release notes
- privacy disclosure
- launch website / FAQ copy

### Step 2. Burn down blockers

- Make `build`, `analyze`, and required tests green.
- Fix or explicitly quarantine flaky / non-blocking tests with written rationale.
- Resolve any launch-scope hardcoded strings or locale mismatches.

Required commands:

```bash
xcodebuild build -scheme nudge -destination 'platform=macOS'
xcodebuild analyze -scheme nudge -destination 'platform=macOS'
xcodebuild test -scheme nudge -destination 'platform=macOS' -only-testing:nudgeTests
xcodebuild test -scheme nudge -destination 'platform=macOS' -only-testing:nudgeUITests
```

### Step 3. Run manual release QA

- Clean install launch
- Accessibility permission allow path
- Accessibility permission deny path -> limited mode
- Menu bar runtime state and countdown
- Idle threshold trigger behavior
- Alert recovery on activity return
- Schedule on/off and next-window behavior
- Daily summary stats visibility
- Settings open / update / relaunch persistence
- KR/EN surface check with screenshots

Evidence:

- screenshots
- screen recording where timing matters
- short test log summary

### Step 4. Lock release assets

- Freeze:
  - `docs/release/v0.1.0.md`
  - README release summary
  - privacy disclosure copy
  - KR/EN screenshots
  - known issues note
- Confirm version metadata remains `0.1.0 (1)` in Release build.

If App Store submission is prepared in parallel:

- replace `beta` wording with `Early Access` or `Preview`
- verify metadata length and privacy wording
- ensure no claim exceeds actual shipping scope

### Step 5. Prepare publish package

- Tag release candidate branch or commit
- Attach:
  - release notes
  - installation / build instructions
  - known issues
  - feedback / issue link
- Decide one public entrypoint:
  - GitHub release page
  - launch website
  - both

### Step 6. Publish and hold

- Publish only after final sign-off
- Keep a promotion hold window so launch posts can be paused if a critical regression is found
- During the first post-release window:
  - monitor launch crash reports / user issues
  - monitor permission confusion reports
  - monitor false-positive / alert-fatigue reports

## 9. Hold and Rollback

- If critical launch regression is found before promotion:
  - do not post public launch messages
  - keep release unpublished or mark it as pre-release
  - continue distributing only to internal testers
- If regression is found after promotion:
  - pin known issue notice
  - pause promotion
  - ship `0.1.1` or replace pre-release artifact

`v0.1.0` is ready to publish only when:

- scope is frozen
- blockers are zero
- required commands are green
- manual release QA is complete
- KR/EN evidence is collected
- support / feedback path is live
