## Findings

### [Resolved] [W-01] `scheduleAlertSync()` — Unbounded Task Spawning (MEDIUM)

**File:** `nudge/Services/IdleMonitor.swift:510-515`
**Type:** Resource Exhaustion / Memory

```swift
private func scheduleAlertSync() {
    let snapshot = runtimeStateController.snapshot
    Task { @MainActor [alertManager] in
        alertManager?.handle(snapshot: snapshot)
    }
}
```

**Risk:** This method is called from **every single state transition** — `recordInput`, `fireIdleDeadline`, `fireAlertEscalationDeadline`, `fireCooldownExpired`, `handleSystemEvent`, `checkSchedule`, `setWhitelistMatched`, `refreshPermission`, `setManualPause`, and more. Each call spawns a new `Task`. Under rapid input events (e.g., user typing fast), this can create a burst of concurrent Tasks all calling `alertManager.handle(snapshot:)` with potentially stale snapshots.

**Impact:** 
- Short-term: CPU spike from concurrent alert handling
- The `AlertManager.handle()` is idempotent enough that correctness isn't broken, but redundant work is performed

**Recommendation:** Consider debouncing or coalescing. A simple `DispatchWorkItem` pattern (already used for deadlines) could replace the fire-and-forget `Task` spawning:

```swift
private var alertSyncWorkItem: DispatchWorkItem?

private func scheduleAlertSync() {
    alertSyncWorkItem?.cancel()
    let workItem = DispatchWorkItem { [weak self] in
        Task { @MainActor [weak self] in
            self?.alertManager?.handle(snapshot: self?.runtimeStateController.snapshot ?? .init())
        }
    }
    alertSyncWorkItem = workItem
    DispatchQueue.main.asyncAfter(deadline: .now() + 0.05, execute: workItem)
}
```

**Resolution:** `scheduleAlertSync()` now coalesces rapid transitions into a single cancellable `DispatchWorkItem` against the latest snapshot.

### [Resolved] [W-02] `AlertSoundPlayer.play()` — Accumulating DispatchWorkItems Without Cleanup on `stopAll()` Race (MEDIUM)

**File:** `nudge/Services/AlertManager.swift:44-54`
**Type:** Memory / Resource

```swift
func play(named soundName: String, repeatCount: Int, interval: TimeInterval) {
    stopAll()
    guard repeatCount > 0 else { return }

    for index in 0..<repeatCount {
        let workItem = DispatchWorkItem { [weak self] in
            self?.playNow(named: soundName)
        }
        pendingWorkItems.append(workItem)
        DispatchQueue.main.asyncAfter(deadline: .now() + (interval * Double(index)), execute: workItem)
    }
}
```

**Risk:** `stopAll()` cancels work items and clears the array, but already-scheduled items on the DispatchQueue may still execute their closure body (cancellation only prevents the block from starting if it hasn't begun). The `[weak self]` capture prevents retain cycles, but if `stopAll()` is called between scheduling and execution, the `pendingWorkItems` array is cleared while the DispatchQueue still holds references to the work items.

**Impact:** Minimal — `weak self` prevents leaks, and `playNow` has its own guard. However, `activeSounds` dictionary could accumulate entries if `stopAll()` is called while sounds are playing, since the cleanup closure in `playNow` (line 75-78) still runs after `stopAll()` clears `activeSounds`.

**Resolution:** `AlertSoundPlayer` now uses a generation counter so stale scheduled repeats and cleanup callbacks are ignored after a later `play()` or `stopAll()`.

```swift
private var currentGeneration = 0

func play(...) {
    currentGeneration += 1
    let generation = currentGeneration
    // In playNow: guard generation == currentGeneration else { return }
}
```

### [Resolved] [W-03] `MenuBarViewModel.refreshMenuSnapshot()` — Double `applySettings` Call Chain (MEDIUM)

**File:** `nudge/Services/MenuBarViewModel.swift:214-229`
**Type:** State Collision / Redundant Work

```swift
func refreshMenuSnapshot(now: Date = .now) {
    let settings = try? modelContext.fetch(FetchDescriptor<UserSettings>()).first
    // ... fetches ...
    
    if let settings {
        idleMonitor.applySettings(settings, at: now)  // ← First apply
    }
    applyMenuSettingsSnapshot(settings)
    applyPetSnapshot(petState)
    idleMonitor.applyWhitelistApps(whitelistApps, at: now)  // ← Triggers whitelist check
}
```

**Risk:** `applySettings` at line 224 already calls `checkSchedule` and potentially `scheduleIdleDeadline`. Then `applyWhitelistApps` at line 228 calls `updateWhitelistMatch`, which calls `setWhitelistMatched`, which may call `sessionTracker.beginSession()`, which fires `onSessionUpdated`, which calls `refreshMenuSnapshot()` again.

**Mitigation (Already Present):** The code at `IdleMonitor.swift:224` has an explicit guard:
```swift
guard runtimeStateController.snapshot.whitelistMatched != matched else { return }
```
This prevents the infinite re-entry loop. The comment at lines 217-223 documents this as "CRITICAL NON-REGRESSION."

**Verdict:** Protected. The unchanged-unmatched whitelist path is now also covered by unit tests.

### [Resolved] [W-04] `RuntimeStateController.transitionLog` — Unbounded Growth (WARNING)

**File:** `nudge/Services/RuntimeStateController.swift:212-224`
**Type:** Memory Leak (Long-running)

```swift
@MainActor
@Observable
final class RuntimeStateController {
    private(set) var snapshot: RuntimeSnapshot
    private(set) var transitionLog: [RuntimeTransitionLogEntry]  // ← Never trimmed
```

**Risk:** Every state transition appends to `transitionLog`. Over a full workday with frequent input events, whitelist switches, schedule checks, and alert escalations, this array can grow to thousands of entries. It is never read or trimmed anywhere in the codebase.

**Impact:** Memory grows linearly with app uptime. For a 10-hour session with ~1 event/second, that's ~36,000 entries. Each entry is ~64 bytes → ~2.3 MB. Not critical, but unnecessary.

**Resolution:** `transitionLog` is now capped and trimmed once it crosses a soft limit.

```swift
func handle(_ event: NudgeWhipRuntimeEvent, at date: Date = .now) {
    snapshot = RuntimeStateReducer.reduce(snapshot, event: event, at: date)
    transitionLog.append(RuntimeTransitionLogEntry(event: event, snapshot: snapshot, occurredAt: date))
    if transitionLog.count > 1000 {
        transitionLog.removeFirst(500)
    }
}
```

### [W-05] `CountdownOverlayController.deinit` — NotificationCenter Observer Leak Risk (LOW)

**File:** `nudge/Views/CountdownOverlayController.swift:30-34`
**Type:** Memory Leak

```swift
deinit {
    for observer in observers {
        NotificationCenter.default.removeObserver(observer)
    }
}
```

**Risk:** `CountdownOverlayController` is a non-`@MainActor` class that holds an `NSPanel` with a `NSHostingView`. The `deinit` runs on whatever thread triggers deallocation. `NotificationCenter.removeObserver` is thread-safe, but the class is initialized on `@MainActor` (from `NudgeWhipAppController`). If deallocation happens off the main thread, there could be a race with the observation callback.

**Impact:** Very low. The observer is for `NSApplication.didChangeScreenParametersNotification`, which fires infrequently.

### [Resolved] [W-06] `SystemEventMonitor` — Global + Local Monitor Double-Fire (LOW)

**File:** `nudge/Services/EventMonitor.swift:45-82`
**Type:** Redundant Processing

```swift
globalMonitor = NSEvent.addGlobalMonitorForEvents(matching: eventMask) { event in
    // ...
}

localMonitor = NSEvent.addLocalMonitorForEvents(matching: eventMask) { event in
    // ...
}
```

**Risk:** When the app is active and has a key window, both global and local monitors fire for the same event. The `shouldSuppressGlobalMenuTrackingDuplicate` method (line 122-130) only suppresses duplicates within `0.1s` of a local menu-tracking event, not for normal double-fires.

**Impact:** `onActivity()` is called twice for each input event when app is active. This triggers `recordInput` → `scheduleIdleDeadline` (cancel + reschedule) → `scheduleAlertSync` (Task spawn) twice. The `scheduleIdleDeadline` method cancels the previous work item before scheduling, so the deadline itself isn't duplicated. But the redundant `scheduleAlertSync()` Task spawns are wasted.

**Resolution:** global monitor events are now ignored while the app is active and already has an active window, leaving local monitor handling as the single in-app event path.

### [W-07] `AlertManager.visualAlertTimestamps` — Pruning Only on `handle()` Call (LOW)

**File:** `nudge/Services/AlertManager.swift:131-133`
**Type:** Memory

```swift
func handle(snapshot: RuntimeSnapshot) {
    let now = nowProvider()
    pruneTimestamps(now: now)  // ← Only pruned when handle() is called
```

**Risk:** If the app enters a long period without any state transitions (e.g., user leaves desk for 4 hours), the timestamp arrays are never pruned. They'll accumulate entries until the next `handle()` call.

**Impact:** Negligible. Even 1000 timestamps at 8 bytes each = 8KB.

### [I-02] `try?` Silently Swallows SwiftData Errors (INFO)

**Multiple files**
**Type:** Error Handling

Throughout the codebase, `try? modelContext.save()` and `try? modelContext.fetch(...)` silently discard errors. In production, if SwiftData encounters a corruption or migration issue, the app will continue running with stale/missing data without any logging or user notification.

**Files affected:**
- `SessionTracker.swift:33, 43, 54, 62, 72`
- `MenuBarViewModel.swift:215-218, 232, 240, 248, 258`
- `NudgeWhipAppController.swift:71, 75, 125`
- `SettingsViewModel.swift:36-37, 47`

**Recommendation:** At minimum, log errors:
```swift
do {
    try modelContext.save()
} catch {
    Logger.swiftData.error("Save failed: \(error)")
}
```

### [I-03] `RuntimeTransitionLogEntry` Not `Sendable` but Used in `@Observable` Context (INFO)

**File:** `nudge/Services/RuntimeStateController.swift:65-69`
**Type:** Thread Safety

```swift
struct RuntimeTransitionLogEntry: Equatable, Sendable {
```

The struct is marked `Sendable`, which is correct. However, `RuntimeStateController` itself is `@MainActor` and `@Observable`. If any code accesses `transitionLog` from a background context (e.g., via a detached Task), it would violate the MainActor isolation. Currently all access appears to be MainActor-safe.

## Positive Findings (What's Done Well)

1. **One-shot deadline pattern** — `IdleMonitor` uses `DispatchWorkItem` with cancel-before-reschedule, not polling loops. This is the correct approach.
2. **`weak self` captures** — All closures in timers and monitors use `[weak self]`, preventing retain cycles.
3. **State machine reducer** — `RuntimeStateReducer.reduce()` is a pure function, making state transitions predictable and testable.
4. **Protocol-based architecture** — All services (`EventMonitoring`, `SystemLifecycleMonitoring`, `FrontmostAppProviding`) are protocol-driven, enabling clean testing.
5. **Explicit re-entry guard** — The whitelist matched guard at `IdleMonitor:224` shows awareness of the infinite loop risk.
6. **`@MainActor` discipline** — All UI-touching and NSEvent-related code is properly isolated to `@MainActor`.
7. **Monitor cleanup** — `SystemEventMonitor.stop()` and `SystemLifecycleMonitor.stop()` properly remove observers and nil out references.

---

## Conclusion

The codebase is **release-ready from a risk perspective**. No infinite loops, no memory leaks that would cause crashes, and no race conditions that would corrupt state. The higher-value hardening items identified in this audit have now been applied.

The remaining notes are informational or tied to intentional beta-scope behavior, not release blockers.
